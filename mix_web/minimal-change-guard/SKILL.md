## When to Use This Skill

当出现以下任意请求时必须使用本 Skill：

- 不要过度设计
- 按最小改动处理
- 不要大范围重构
- 只修当前问题
- 局部修复，不要扩散
- 不要顺手改一堆无关代码
- 先让功能可用，再说优化

## Goal

在满足当前需求的前提下，把改动控制在最小必要范围内，避免为了抽象、统一、未来扩展或“代码更漂亮”而引入额外复杂度。

## Core Principles

### 1. Smallest Viable Change

优先选择“能解决当前问题的最小实现”：

- 优先改已有逻辑，而不是新建一层包装
- 优先补一处判断、一个分支、一个参数，而不是整段重写
- 优先局部修复已有函数，而不是提前拆出多个新类 / 新接口 / 新模式

### 2. YAGNI

不要为“以后可能会用到”提前设计：

- 不预埋暂未被需求证明的扩展点
- 不为了未知的第二个场景提前抽象公共层
- 不为了“未来也许支持更多类型”先引入策略、工厂、配置驱动

### 3. KISS

保持方案直接、可读、易验证：

- 能用现有分支解决，就不要上设计模式
- 能在原文件中清晰完成，就不要拆出多层跳转
- 能靠现有数据结构表达，就不要新增中间模型

### 4. Reuse Existing Patterns First

先复用当前模块已经存在的实现方式：

- 优先沿用现有目录结构、命名方式、调用链路
- 优先跟随同文件、同模块、同层已有写法
- 若代码库已经有约定俗成的模式，优先兼容它，而不是替换它

### 5. Pragmatic DRY

避免机械追求“零重复”：

- 少量、稳定、易懂的重复，通常优于错误抽象
- 只有当重复已经明确出现并且抽取后更简单时，才做复用
- 不要为了合并两三行相似代码，引入更难理解的通用层

### 6. Pragmatic SOLID

SOLID 是参考，不是强制重构理由：

- 只有在现有改动已经明显被职责混乱、耦合过高阻塞时，才做必要整理
- 若当前需求不需要，不主动拆接口、拆服务、拆基类
- 不为了“理论上更合理”而扩大改动面

## Workflow

### Step 1: 只读必要上下文

先确认“为了完成这次需求，最少需要读哪些文件”：

1. 直接报错或直接需求涉及的文件
2. 该文件的直接调用方或被调用方
3. 当前改动必须依赖的测试、配置或类型定义

禁止一上来全仓扫描、全链路重构式分析。

### Step 2: 先给出最小改动方案

在动手前先做如下判断：

1. 是否能在原位置直接修复？
2. 是否能复用已有函数、类、配置、常量？
3. 是否真的需要新增文件、抽象层、公共方法？

如果前两项可以成立，默认不进入第 3 项。

### Step 3: 默认就地修改

优先级必须如下：

1. 修改现有实现
2. 在现有模块内新增极小辅助函数
3. 在现有目录下新增小型配套文件
4. 新增抽象层、通用层、跨模块重构

只有当前 1-3 明显无法合理解决问题时，才考虑第 4 项。

### Step 4: 超出原请求范围时先确认

如果出现以下任一情况，必须先明确提示用户，再决定是否扩大改动：

- 需要顺手重构 unrelated code
- 需要移动文件或改公共接口
- 需要批量调整命名、目录结构、基础设施
- 需要修改多个模块才能“做得更优雅”
- 当前问题的真正修复会影响较大行为边界

不能把“顺便优化一下”当成默认动作。

### Step 5: 交付前做边界自检

提交结果前至少检查：

- 是否只改了和当前需求强相关的文件？
- 是否引入了不必要的新抽象？
- 是否为了统一风格修改了无关代码？
- 是否能删掉一部分新增代码后仍满足需求？

如果答案是“能”，继续收缩改动。

## Scope Rules

### 默认允许

- 修复当前 bug
- 给现有逻辑补条件、补参数、补校验
- 在原测试文件中补充针对性用例
- 沿用已有模式做小范围扩展

### 需要非常谨慎

- 提炼公共方法
- 新增 helper / util
- 调整文件职责
- 扩大测试覆盖到不相关路径

只有在能明显减少当前实现复杂度时才做，且必须控制在当前模块内。

### 默认禁止

- 无需求依据的架构升级
- 为未来场景预留大量扩展点
- 把局部修复升级为全局重构
- 因“代码味道”而顺手清扫大面积旧代码
- 重写已稳定且本次需求未触及的模块

## Output Requirements

使用本 Skill 时，回复中应明确体现以下判断：

1. 当前按“最小改动原则”处理
2. 改动限制在哪些文件 / 模块
3. 哪些“可以做但这次故意不做”的扩展被主动排除
4. 若确实需要扩大范围，要说明触发原因

## Recommended Phrases

适合直接对用户说明的表达：

- “这次按最小改动处理，只在现有链路上修复。”
- “我先复用当前模块已有模式，不额外引入新抽象。”
- “这个问题可以局部解决，暂时不建议上升为通用重构。”
- “如果要继续做成更通用方案，需要扩大改动范围，我会先单独说明。”

## Constraints

- 不要把“工程美感”置于“完成当前需求”之前
- 不要把“主流设计思想”理解成“必须更多抽象”
- 所有新增层次都必须能回答清楚：为什么当前需求非它不可？
- 如果局部方案已经足够清晰可靠，就不要继续优化成通用方案

## Checklist

- [ ] 已确认最小必要修改点
- [ ] 已优先复用现有实现
- [ ] 未引入无需求依据的新抽象
- [ ] 改动范围未扩散到无关模块
- [ ] 若必须扩大范围，已先向用户说明原因
