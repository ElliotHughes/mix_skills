## Execution Contract

If this skill is triggered, the assistant MUST begin the response with:

=== USING SKILL: requirement-doc-to-design ===

---

## When to Use This Skill

当出现以下任意请求时必须使用本 Skill：

- 把 `docs/需求文档/*.md` 转成技术设计文档
- 基于需求文档输出可实施方案、接口设计、错误码设计、service 设计
- 给前端和测试补联调文档、验收文档、回归清单
- 需求涉及接口字段、错误码、全局变量、发布与回滚说明
- 需要把需求文档中的“技术设计 / 验收文档”段落回填成实际文档链接

## Goal

在 `mix_web` 仓库内，基于需求文档和现有代码，稳定产出以下内容：

1. 后端技术设计文档
2. 必要时的前端 / 测试联调文档
3. 需求文档中的技术设计与验收文档入口回填

## Workflow

### Step 1: 先读需求，再补代码上下文

至少读取以下内容：

1. 目标需求文档
2. 对应 controller / service / validator / repository / model
3. `response/ErrorCode.php`
4. 配置来源，例如 `SysGlobalVarModel`、公司配置、常量配置
5. `docs/设计文档` 下现有同类文档，用来对齐风格

原则：

- 只读和当前需求强相关的文件，不要无差别扫全仓库
- 优先确认“现在怎么做”，再写“准备怎么改”
- 如果代码里已有接口或能力，优先写“复用 / 扩展”，不要凭空重造一套

### Step 2: 判断要产出几份文档

默认至少产出 1 份后端技术设计文档。

满足以下任一条件时，再补 1 份前端 / 测试联调文档：

- 接口请求参数、响应字段、错误码有变化
- 前端页面状态、弹窗、标签展示依赖后端返回
- 测试需要明确回归范围、边界用例、配置检查项
- 需求涉及新增全局变量、发版依赖或回滚注意事项
- 用户明确要求“给前端 / 测试同学看的文档”

### Step 3: 输出后端技术设计文档

建议命名：

`docs/设计文档/YYYY-MM-DD_<TOPIC>_TECH_DESIGN.md`

文档至少覆盖以下章节：

1. 文档信息
2. 背景与目标
3. 现状与约束
4. 方案概览
5. 接口设计
6. 错误码设计
7. 配置 / 全局变量设计（如果有）
8. Service / Repository 设计
9. 数据库与迁移说明
10. 回归点与测试建议
11. 需求估时

写法要求：

- 以中文为主
- 错误 message 示例必须用 en-US
- 接口要写清 URL、Method、请求、响应、兼容性
- 估时统一按 `7.5h = 1 人日`
- 明确哪些接口不变，不要把“保持不变”的能力也误写成要改造

### Step 4: 输出前端 / 测试联调文档

建议命名：

`docs/设计文档/YYYY-MM-DD_<TOPIC>_FE_QA_GUIDE.md`

文档至少覆盖以下内容：

1. 本次影响哪些接口，不影响哪些接口
2. 前端需要关注的新字段、新错误码、页面表现
3. 前端不需要改的部分
4. 测试需要覆盖的核心用例、边界用例、回归范围
5. 配置与数据检查项
6. 上线发版顺序
7. 发版注意事项
8. 回滚注意事项
9. 结果判断标准

"上线发版顺序"章节必须写清以下默认顺序：

1. 先跑 SQL（含表结构变更、数据迁移脚本等）
2. 再发后端（不需要分线路，统一发布即可）
3. 最后发前端

如果本次需求没有 SQL 变更，跳过第 1 步并注明"无 SQL 变更"；如果没有前端变更，跳过第 3 步并注明"无前端变更"。

特别注意：

- 如果需求引入了新的全局变量，例如 `sys_global_var` key，测试文档必须写清：
  - key 名
  - 默认值
  - 缺失时的兜底行为
  - 非法值时的兜底行为
  - 发版是否依赖先加配置
  - 回滚时配置是否可以保留
- 如果前端会根据错误码弹窗，优先让前端按错误码走本地化，不直接使用后端英文 message

### Step 5: 回填需求文档

完成文档后，回填原需求文档：

- `技术设计` 段落填技术设计文档路径
- `验收文档 / 资料` 段落填 FE/QA 文档路径（如果有）

要求：

- 不要让需求文档继续保留“请补充技术设计文档”这类占位语
- 回填路径要和实际文件一致

### Step 6: 手工校验

至少检查以下内容：

- 文档路径真实存在
- 接口名、控制器名、类名、错误码名与代码一致
- 新增配置项名称前后一致
- 发版与回滚说明没有互相冲突
- 如果本次只是文档任务，`git diff` 应主要集中在 `docs/` 或 skill 仓库对应目录

## 推荐写法

### 技术设计文档的常见结论

优先写清：

- 现有逻辑已经支持什么
- 本次真正新增的是什么
- 新逻辑应该放在哪一层
- 哪些地方需要前后端联动
- 哪些场景需要后端兜底，不能只靠前端判断

### FE / QA 文档的常见结论

优先写清：

- 前端新增依赖的字段和错误码
- 测试需要准备的账号 / 配置 / 数据状态
- 配置缺失、非法值、回滚等非功能场景
- 上线发版顺序（默认：SQL → 后端 → 前端）
- 发布时除了代码之外，还需要同步检查什么

## Constraints

- 优先复用现有 `docs/设计文档` 的文风和结构，不要另起一套完全不同的模板
- 没有表结构改动时，要明确写“无表结构变更”，不要模糊处理
- 没有新增接口时，可以写“修改接口”或“接口保持不变，仅扩展字段 / 错误码”
- 不要发明需求文档和代码里都不存在的业务规则；如果必须做推断，要明确标注“建议”或“推断”
- 不要把面向后端实现的细节全部塞进 FE / QA 文档；那份文档要以联调和验收为中心

## Checklist

- [ ] 技术设计文档已创建
- [ ] FE / QA 联调文档在需要时已创建
- [ ] 需求文档入口已回填
- [ ] 接口变更、错误码、配置项、回归点、估时已覆盖
- [ ] 如涉及全局变量，发版与回滚说明已覆盖
